# API Documentation

## Overview

**Module:** `api_doc`
**Category:** Hidden
**Version:** 1.0
**Depends:** `web`
**Summary:** Dynamic, runtime-generated API documentation page with interactive playground for executing Odoo model methods over HTTP
**Auto-install:** `True`
**Bootstrap:** `True`
**License:** LGPL-3
**Author:** Odoo S.A.

`api_doc` is a developer-focused module that provides a single-page application (SPA) at the `/doc` URL. It introspects the Odoo registry at runtime to generate documentation for all installed models, their fields, and their methods. It also includes an interactive API playground that lets developers execute Odoo model methods directly from the browser, with auto-generated code examples in Python, JavaScript, cURL, and JSON. Because it reads the live database registry rather than parsing source files, the documentation is always up to date with the actual installed version.

### Key Design Decisions

- **Runtime introspection:** Documentation is generated by querying `ir.model` and `ir.model.fields` records, not by parsing Python source. This means documentation is always current with the installed modules and reflects any custom fields added in the database.
- **Access-controlled:** The `/doc` endpoint checks `api_doc.group_allow_doc` access group. Only users in this group (typically developers and administrators) can access the documentation and the playground.
- **Cached index:** The `/doc/index.json` endpoint generates a large JSON index (often >1 MiB with many modules). To avoid regenerating it on every request, the index is stored as an `ir.attachment` and served with an ETag. The garbage collection method `_gc_doc_index` removes stale cached indexes when the database registry sequence changes.
- **JSON2 request type:** The API endpoints use `type='json2'` (Odoo's JSON-RPC-like format over HTTP POST), which provides a cleaner request/response format than standard JSON-RPC while supporting auth modes like `auth='bearer'` and `auth='user'`.
- **OWL SPA architecture:** The frontend is built as an OWL (Odoo Web Library) single-page application, with separate components for sidebar navigation, model details, method documentation, and the request playground. The `DocClient` root component manages global state and routing.

## Backend Controller

### `DocController` — API Documentation Endpoints

**File:** `controllers/api_doc.py`

The controller provides three HTTP endpoints and one JSON-RPC endpoint.

#### `GET /doc` — Documentation SPA

```python
@http.route(['/doc', '/doc/<model_name>', '/doc/index.html'], type='http', auth='user')
def doc_client(self, mod=None, **kwargs):
    if not self.env.user.has_group('api_doc.group_allow_doc'):
        raise AccessError(self.env._(
            "This page is only accessible to %s users.",
            self.env.ref('api_doc.group_allow_doc').sudo().name))
    res = request.render('api_doc.docclient')
    res.headers['X-Frame-Options'] = 'deny'
    return res
```

Renders the OWL SPA (`api_doc.docclient` template). Supports direct deep linking to a specific model via `/doc/<model_name>`. The `X-Frame-Options: deny` header prevents embedding in iframes for security.

#### `GET /doc/index.json` — Module and Model Index

```python
@http.route('/doc/index.json', type='json2', auth='user')
def doc_index(self):
    """Returns: {'modules': [...], 'models': [{'model': str, 'name': str, 'fields': {...}, 'methods': [...]}]}"""
    # ... cache key computation using HMAC of db_registry_sequence + lang + user groups
    # ... cached in ir.attachment as 'odoo-doc-index-<sequence>-<hmac>.json'
    modules, models = self._doc_index()
    return {'modules': modules, 'models': models}
```

Returns a lightweight index of all accessible models and installed module names. The response is cached in `ir.attachment` to avoid regenerating it on every request. The cache key includes the database registry sequence number, the user's language, and the user's group IDs — ensuring the cache is invalidated when the schema changes, the language changes, or the user's group membership changes (which may affect field access).

#### `GET /doc/<model_name>.json` — Full Model Documentation

```python
@http.route('/doc/<model_name>.json', type='json2', auth='user', readonly=True)
def doc_model(self, model_name):
    """Returns: {'model': str, 'name': str, 'doc': str, 'fields': dict, 'methods': dict}"""
    Model = self.env[model_name]
    ir_model = self.env['ir.model']._get(model_name)
    result = {
        'model': model_name,
        'name': ir_model.name,
        'doc': None,  # TODO
        'fields': {
            field['name']: dict(field, module=next(iter(Model._fields[field['name']]._modules), None))
            for field in Model.fields_get().values()
        },
        'methods': {
            method_name: self._doc_method(Model, model_name, method, method_name)
            for method_name in dir(Model)
            if (method := is_public_method(Model, method_name))
        },
    }
    return result
```

Returns comprehensive documentation for a single model, including:
- All field definitions from `fields_get()` (type, string, help, etc.)
- Module that introduced each field (from `Model._fields[field_name]._modules`)
- All public method signatures with parameter details, return annotations, docstrings, and raised exceptions

#### `_doc_method(Model, model_name, method, method_name)` — Method Reflection

This internal method builds a structured reflection of a model's method using Python's `inspect` module:

```python
def _doc_method(self, model, model_name, method, method_name):
    introducing_class = next(
        parent_class
        for parent_class in reversed(type(model).mro())
        if hasattr(parent_class, method_name)
    )
    signature = parse_signature(introducing_method)
    return signature.as_dict() | {
        'model': introducing_class._name or 'core',
        'module': introducing_class._module or 'core',
    }
```

The method:
1. Finds the class that first introduced this method (by traversing the MRO in reverse)
2. Parses the function signature (parameter names, defaults, annotations, return type)
3. Extracts docstring information using docutils for RST-formatted parameter docs
4. Returns a structured dict with `signature`, `parameters`, `return`, `raises`, `doc`, `api`, `model`, and `module`

### Signature Parsing

**File:** `controllers/api_doc.py` — `parse_signature()` function

The signature parser strips `self` and `cls` from the displayed signature, replaces `BaseModel` return annotations with `list[int]` (since methods typically return recordset IDs), and uses dataclasses to represent `Signature`, `Param`, and `Return` objects. The `as_dict()` method produces a compact JSON representation, omitting default values for non-JSON-serializable objects (e.g., model objects).

### Docstring Enhancement

**File:** `controllers/api_doc.py` — `enhance_signature_using_docstring()` function

Uses `docutils` to parse reStructuredText docstrings and extract `:param`, `:type`, `:returns`, `:raises` fields. These are merged into the signature object, providing rich parameter documentation when present in the source code.

Supported RST field formats:
- `:param <name>:` — parameter description
- `:param <annotation> <name>:` — parameter with type annotation
- `:type <name>:` — parameter type
- `:returns:` or `:return:` — return value description
- `:rtype:` — return type annotation
- `:raises <Exception>:` — exception documentation

### Access Control

All three HTTP endpoints check `api_doc.group_allow_doc` via `has_group()`. The `doc_bearer_index` and `doc_bearer_modec` endpoints (with `type='json2', auth='bearer'`) use API key authentication and have the same access control check.

### `auth='bearer'` Endpoints

```python
@http.route('/doc-bearer/index.json', type='json2', auth='bearer')
def doc_bearer_index(self):
    return self.doc_index()

@http.route('/doc-bearer/<model_name>.json', type='json2', auth='bearer', readonly=True)
def doc_bearer_modec(self, model_name):
    return self.doc_model(model_name)
```

These endpoints support API key authentication (auth='bearer') and are intended for external API consumers who want to programmatically fetch model documentation using a stored API key rather than session auth.

## Backend Model

### `ir.attachment` — Index Garbage Collection

**File:** `models/ir_attachment.py`

```python
class IrAttachment(models.Model):
    _inherit = 'ir.attachment'

    @api.autovacuum
    def _gc_doc_index(self):
        """ Garbage collect the outdated /doc/index.json attachments. """
        sequence = str(self.env.registry.get_sequences(self.env.cr)[0])
        attachments = self.search_fetch(
            [('name', 'like', R'odoo-doc-index-%-%.json')],
            ['name'],
        ).filtered(
            lambda doc: doc.name.split('-')[3] != sequence,
        )
        if attachments:
            attachments.unlink()
```

| Aspect | Detail |
|---|---|
| Trigger | `@api.autovacuum` — runs on Odoo startup and periodically |
| Purpose | Remove stale documentation index caches |
| Match condition | Attachment name matches `odoo-doc-index-%-%.json` |
| Retention | Keeps only the index whose sequence number matches the current DB registry sequence |
| Naming format | `odoo-doc-index-<sequence>-<hmac>.json` |

The sequence number in the filename encodes the database schema version. When a module is installed, upgraded, or a field is added, the sequence increments, automatically orphaning the old cached index.

## Frontend Architecture

### Root Component: `DocClient`

**File:** `static/src/doc_client.js`

The `DocClient` component is the SPA root. It:
- Sets the UI theme (`odoo-dark` or `odoo-light`) from localStorage on mount
- Initializes the `ModelStore` and `useDocUI` stores
- Loads the model index via `modelStore.loadModels()` on mount
- Handles URL routing for deep-linked models (`/doc/<model_name>`) and method/field anchors (`/doc/<model_name>#<method_name>`)
- Listens to `popstate` events to keep URL state in sync with UI state

### `ModelStore`

**File:** `static/src/doc_model_store.js`

Manages the in-memory model index and fetches detailed model documentation on demand. The store:
- Loads the lightweight index from `/doc/index.json` on startup
- Fetches full model details from `/doc/<model_name>.json` on demand
- Caches fetched model data in memory for the session duration
- Tracks active model, method, and field selection

### Code Generation

**File:** `static/src/utils/doc_code_gen.js`

Generates executable code samples in four languages from an API request object:

| Language | Output format |
|---|---|
| JSON | Raw JSON payload with inline domain arrays |
| JavaScript | `fetch()` async/await example using ES modules |
| Python | `requests.post()` example |
| cURL | `curl` command with `--oauth2-bearer` for API key auth |

The generator handles type conversion (Python `None` for JSON `null`, removed quotes from JSON keys) and embeds the correct URL from the current request context.

### `api_action.js` — In-Form Method Executor

**File:** `static/src/api_action.js`
**Asset bundle:** `web.assets_backend` (not in `api_doc.assets`)

This file is loaded separately in the backend web client (not the documentation SPA) and provides the ability to execute model methods directly from a form's action button. It is referenced in the manifest as being removed from the main `api_doc.assets` bundle and loaded in `web.assets_backend` instead, so it does not bloat the documentation page but is available everywhere in the backend.

## Access Control and Security

### Security Groups

**File:** `security/res_groups.xml`

The module defines `api_doc.group_allow_doc`. Users must belong to this group to access:
- The `/doc` page (HTML SPA)
- The `/doc/index.json` and `/doc/<model_name>.json` endpoints
- The API playground

This prevents non-developer users from viewing model internals and executing arbitrary methods.

### XSS Protection

The HTML documentation pages use Odoo's template rendering engine (QWeb), which outputs HTML-escaped content by default. The `X-Frame-Options: deny` header prevents clickjacking attacks.

### Method Execution

The playground allows executing arbitrary model methods on the server. This is intentional for developer use but should be restricted to trusted users only via access group management.

## Performance Notes

- The index endpoint is cached in `ir.attachment` to avoid repeated generation of the large JSON payload.
- Model detail endpoints are not cached server-side (they use `no-cache` headers) but the browser can cache them using the ETag.
- The `_gc_doc_index` autovacuum method runs on startup and keeps the attachment table clean.
- The OWL SPA loads model data on demand, avoiding the need to fetch all model details upfront.

## Migration Notes

Key changes for migration:

- The `auth='bearer'` endpoints for API key auth were introduced in Odoo 19. Earlier versions may not support this auth mode.
- The `json2` request type is an Odoo 19 enhancement over the older `json` type. Ensure any custom endpoints use the correct type.
- The `fields_get()` response structure and `Model._fields` introspection remain compatible, but custom field types introduced by community modules may not have complete type metadata.
- If upgrading from a version without HMAC-cached indexes, the old cache entries will be cleaned up by `_gc_doc_index` on the first startup after upgrade.
