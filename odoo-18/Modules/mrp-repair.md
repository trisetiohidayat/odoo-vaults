# mrp_repair — Repair Management (MRP Extension)

**Module:** `mrp_repair`
**Odoo Version:** 18
**Source:** `~/odoo/odoo18/odoo/addons/mrp_repair/`
**Depends:** `repair`, `mrp`
**Type:** Integration (MRP + Repair + Stock)
**Auto-install:** True (with `repair` and `mrp`)

---

## Purpose

The `mrp_repair` module bridges the **repair** module and the **MRP** module. It adds two capabilities to the base repair workflow:

1. **Manufacturing Order Generation** from a repair order — a repair order can trigger creation of an MO via the procurement system (for example, manufacturing a replacement component as part of the repair).
2. **MRP Visibility into Repair Source** — a manufacturing order can display its source repair orders, closing the loop for companies that track which MOs were created to support repairs.

Additionally, it handles **phantom BoM explosion**: if a part added to a repair order is a kit/product with a phantom BoM, the module automatically explodes the kit into its component moves at repair order create/write time.

The base `repair.order` model is defined in `addons/repair/models/repair.py`. `mrp_repair` extends it with:

|-------|------|-------------|

### Key Methods (mrp_repair Extension)

#### `action_explode()`

**Location:** `models/repair.py`

Called automatically on `create` and `write` of a repair order. This is the **phantom BoM explosion** mechanism.

```
For each stock.move in repair.move_ids:
  1. Find phantom BoM for the move's product via MrpBom._bom_find(..., bom_type='phantom')
  2. If no phantom BoM: skip
  3. Compute factor = (product_uom_qty / bom.product_qty) × bom.product_uom conversion
  4. Call bom.sudo().explode(product_id, factor) → returns (boms, lines)
  5. For each non-service BoM line:
       - Create phantom line via op._prepare_phantom_line_vals(bom_line, qty)
  6. Delete the original phantom move
  7. Create the exploded component moves
```

**L4 Note — Phantom BoM Explosion:**
- This ensures that when a user adds a kit product (e.g., "Computer Kit") as a repair part, the system automatically resolves it into individual components (motherboard, RAM, SSD, etc.) as separate `stock.move` records linked to the repair order.
- Explosion runs with `sudo()` for both the BoM search and the `explode()` call, then unlinks/creates as superuser.
- Original moves are replaced, not updated — old moves are deleted and new ones created.
- This only applies to `stock.move` records where `repair_line_type` is set (i.e., parts lines, not the product-to-repair move itself).

#### `_prepare_phantom_line_vals(bom_line, qty)`

**Location:** `models/repair.py` (on `stock.move`)

Called by `action_explode()` to prepare values for a phantom component move:

```python
{
    'name': self.name,                        # Repair order reference
    'repair_id': self.repair_id.id,
    'repair_line_type': self.repair_line_type, # 'add' / 'remove' / 'recycle'
    'product_id': bom_line.product_id.id,
    'price_unit': self.price_unit,            # Inherit price from parent
    'product_uom_qty': qty,                   # Exploded quantity
    'location_id': self.location_id.id,
    'location_dest_id': self.location_dest_id.id,
    'state': 'draft',
}
```

#### `action_view_mrp_productions()`

Returns an `ir.actions.act_window` for the related MOs:
- **1 MO** → opens form view directly (`res_id` set)
- **Multiple MOs** → opens list view with domain `[('id', 'in', production_order_ids.ids)]`
- **0 MOs** → returns empty action (no navigation)

The MOs are found via `procurement_group_id.stock_move_ids.created_production_id`.

### Location Routing for Repair Moves

The base `repair` module defines the location routing map:

```
MAP_REPAIR_LINE_TYPE_TO_MOVE_LOCATIONS_FROM_REPAIR = {
    'add':     {location_id: 'location_id',            location_dest_id: 'location_dest_id'},
    'remove':  {location_id: 'location_dest_id',       location_dest_id: 'parts_location_id'},
    'recycle': {location_id: 'location_dest_id',       location_dest_id: 'recycle_location_id'},
}
```

- **Add** — Components consumed from inventory (`location_id` = warehouse stock) → `location_dest_id` (repair staging area). Consumed from stock.
- **Remove** — Parts taken off the product go to the removed-parts location.
- **Recycle** — Parts go to the recycling location.

### Stock Move Methods Added by `repair` Module

|--------|-------------|

| `_compute_location_dest_id()` | Routes `location_dest_id` based on `repair_line_type` |
| `_is_consuming()` | Returns `True` if `repair_line_type == 'add'` |
| `_set_repair_locations()` | Syncs location when repair order locations change |
| `_create_repair_sale_order_line()` | Creates SOL for 'add' type moves when repair has linked SO |
| `_clean_repair_sale_order_line()` | Sets SOL qty to 0 on cancel |
| `_update_repair_sale_order_line()` | Syncs SOL qty when move qty changes |

| `_split()` | Prevents move splitting for repair moves with partial qty |
| `_get_source_document()` | Returns `repair_id` for repair moves |

### `action_explode()` in `mrp_repair` Context

When `action_explode()` runs (on create/write of `repair.order`), it calls `op._prepare_phantom_line_vals()` which sets `repair_id` and `repair_line_type` on the new component moves. These new moves are created in `state='draft'`.

Additionally, `stock.move` in the base repair module (`models/stock_move.py`) overrides `_prepare_phantom_move_values()` to carry `repair_id` forward:

```python
def _prepare_phantom_move_values(self, bom_line, product_qty, quantity_done):
    vals = super()._prepare_phantom_move_values(...)
    if self.repair_id:
        vals['repair_id'] = self.repair_id.id
    return vals
```

This ensures that if a phantom kit is exploded from within a repair move (at the base stock.move level), the `repair_id` is preserved.

## Stock Picking Integration

`repair.order` does **not** inherit from `stock.picking`. Instead it uses `stock.picking.type` with `code='repair_operation'` and manages its own `stock.move` records independently. The `procurement_group_id` links the repair to any downstream manufacturing orders it triggers.

### Repair Order ↔ Stock Picking Comparison

| Aspect | `repair.order` | `stock.picking` |
|--------|----------------|-----------------|
| Inherits | `mail.thread`, `product.catalog.mixin` | `mail.thread`, `stock.picking` |
| State machine | `draft → confirmed → under_repair → done → cancel` | `draft → confirmed → assigned → done → cancel` |
| Move lines | Via `stock.move` (not picking-bound) | Via `stock.move` |
| Picking type | `repair_operation` | any type |
| SO binding | `sale_order_id` + `sale_order_line_id` | `sale_id` |

## Sale Order Integration (via `repair` module)

### From Sale Order to Repair Order

When a sale order line has `product_template.create_repair = True`:
1. SO confirmation (`_action_confirm`) calls `_create_repair_order()` on SOLs
2. For serial-tracked products: one RO per unit (`qty = 1`, repeated `product_uom_qty` times)
3. For consumable products: one RO with `product_qty = line.product_uom_qty`
4. RO is created in `state='confirmed'` with `sale_order_id` and `sale_order_line_id` set

### From Repair Order to Sale Order

`repair.order.action_create_sale_order()`:
1. Creates a new `sale.order` linked to the repair's `partner_id` and `warehouse_id`
2. Calls `move_ids._create_repair_sale_order_line()` for each `'add'` type move
3. SOLs get `price_unit = move.price_unit`, or `0.0` if `under_warranty = True`

## Warranty Processing (L4)

**`under_warranty` field** on `repair.order`:
- Boolean, defaults to `False`
- When `True`: sets `price_unit = 0.0` on all SOLs created from repair 'add' moves
- When toggled on an existing repair: `_update_sale_order_line_price()` zeroes out prices
- When toggled off: `_compute_price_unit()` recomputes from product list price

Warranty repairs do not generate invoices — they are cost-only from the company's perspective (absorbed as warranty expense via the expense account).

---

## Invoicing Flow (L4)

Repair orders have **no direct invoice creation** — invoicing flows through the sale order:

```
repair.order (under_warranty=False)
  └── sale_order_id
        └── sale.order.line (price_unit = move.price_unit)
              └── sale.order.invoice (standard SO → INV flow)

repair.order (under_warranty=True)
  └── No invoice — cost absorbed by company
```

The `repair.fee` concept from older Odoo versions is replaced by `stock.move` records of type `'add'` — their `price_unit` becomes the SOL price, which is then invoiced through the standard SO workflow.

---

## Stock Parts Consumption (L4)

When a repair order moves to `done`:

1. `action_repair_done()` creates a **final product move** (`repair.move_id`) moving the repaired product from `product_location_src_id` to `product_location_dest_id`
2. All `'add'` type moves (component consumption) are marked `picked = True` if not already
3. All moves (`move_ids` + the final product move) call `_action_done(cancel_backorder=True)`
4. Stock quants are updated: components decrease at `location_id`, final product increases at `product_location_dest_id`

Removed parts (type `'remove'`) go to `parts_location_id`. Recycled parts go to `recycle_location_id`.
