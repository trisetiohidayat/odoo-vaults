# auth_passkey — Passkey / WebAuthn Authentication

> Implements the W3C WebAuthn standard for passwordless, biometric authentication. Users register cryptographic passkeys (stored in OS credential managers, hardware keys, or cloud sync) and use them to log into Odoo. When authenticating via passkey, MFA is automatically skipped.

**Source:** `~/odoo/odoo18/odoo/addons/auth_passkey/`
**Manifest:** `__manifest__.py`
**Version:** `1.0`
**Depends:** `base_setup`, `web`
**License:** LGPL-3
**Installable:** `True`

---

## Overview

Passkeys in Odoo 18 implement the **WebAuthn Level 1/2/3** specification (FIDO2). The key cryptographic material is:

- **Credential ID** — a unique public identifier for the registered authenticator, stored in the Odoo DB
- **Public Key** — the asymmetric key pair generated by the authenticator; the private key never leaves the device
- **Sign Count** — a monotonically increasing counter used by the authenticator to prevent replay attacks

The module uses a vendored WebAuthn library (`_vendor/webauthn/`) rather than an external PyPI dependency, ensuring full control over the implementation.

## Models

### `auth.passkey.key` — Passkey Registration Record

Stores one registered passkey per user per device. The record links a user's Odoo account to a registered WebAuthn credential.

**`_name = 'auth.passkey.key'`**
**`_description = 'Passkey'`**
**`_order = 'id desc'`**

#### Fields

| Field | Type | Groups | Description |
|-------|------|--------|-------------|
| `name` | `Char` | All | Human-readable name the user gave this passkey (e.g., "MacBook Pro Touch ID", "YubiKey 5"). User-defined at registration time. |
| `credential_identifier` | `Char` | `base.group_system` only | The WebAuthn credential ID (base64url-encoded). Used to look up the passkey during login. Unique per record. |
| `public_key` | `Char` | `base.group_system` only | The credential public key (PEM or base64url-encoded). Used server-side to verify authentication signatures. Stored in a separate DB column via `init()` migration. |
| `sign_count` | `Integer` | `base.group_system` only | The authenticator's current sign count. Incremented after each successful authentication. Used to detect cloned authenticators. Default: `0`. |

#### SQL Constraints

```python
_sql_constraints = [
    ('unique_identifier', 'UNIQUE(credential_identifier)',
     'The credential identifier should be unique.'),
]
```

Enforces that each credential ID is registered only once across all users and all systems.

#### `init()` — Schema Migration

```python
def init(self):
```

Handles the upgrade path: adds the `public_key` column if it does not exist (this column was added in Odoo 18 — prior versions stored the public key differently).

#### Methods

**`_start_registration()`** — `@api.model` — Generate registration options

```python
def _start_registration(self):
    registration_options = json.loads(options_to_json(generate_registration_options(
        rp_id=url_parse(self.get_base_url()).host,
        rp_name='Odoo',
        user_id=str(self.env.user.id).encode(),
        user_name=self.env.user.login,
        authenticator_selection=AuthenticatorSelectionCriteria(
            resident_key=ResidentKeyRequirement.REQUIRED,
            user_verification=UserVerificationRequirement.REQUIRED
        )
    )))
    request.session['webauthn_challenge'] = registration_options['challenge']
    return registration_options
```

**L4: What this does:**
1. Generates a random challenge (cryptographically secure, stored in the session).
2. Returns WebAuthn registration options to the JS frontend.
3. `rp_id` = the domain name of the Odoo instance (e.g., `mycompany.odoo.com`).
4. `resident_key=REQUIRED` — the authenticator must store the credential as a "discoverable credential" (enables username-less login).
5. `user_verification=REQUIRED` — the user must prove their identity (PIN, fingerprint, face) in addition to possessing the key. This makes passkey login equivalent to 2FA — something you have (the key) plus something you know (the PIN/biometric).
6. `user_id` is the Odoo user ID encoded as bytes.

**RP ID rules:** The `rp_id` must match the domain (or be a subdomain) of the site. `localhost` cannot be used in production.

**`_verify_registration_options(registration)`** — `@api.model` — Verify and store new passkey

```python
def _verify_registration_options(self, registration):
    verification = verify_registration_response(
        credential=registration,
        expected_challenge=base64url_to_bytes(self._get_session_challenge()),
        expected_origin=parsed_url.replace(path='').to_url(),
        expected_rp_id=parsed_url.host,
        require_user_verification=True,
    )
    return {
        'credential_id': verification.credential_id,
        'credential_public_key': verification.credential_public_key,
    }
```

Called after the user completes the browser's passkey registration dialog. The browser returns a `registration` object containing:
- `attestationObject` — the authenticator's attestation (proof of what kind of device made the key)
- `clientDataJSON` — browser metadata including the challenge and origin

The server verifies the signature using the trust chain from the attestation (or skips attestation validation if not provided). Then extracts the `credential_id` and `credential_public_key` for storage.

```python
        rp_id=url_parse(self.get_base_url()).host,
```

Called by the JS frontend when the user clicks "Login with Passkey". Unlike registration, authentication options do NOT include `user_id` — this allows the user to select any registered passkey for this RP (the credential ID is returned in the response).

**`_verify_auth(auth, public_key, sign_count)`** — `@api.model` — Verify authentication response

```python
def _verify_auth(self, auth, public_key, sign_count):
    auth_verification = verify_authentication_response(
        credential=auth,
        expected_challenge=base64url_to_bytes(self._get_session_challenge()),
        expected_origin=parsed_url.replace(path='').to_url(),
        expected_rp_id=parsed_url.host,
        credential_public_key=base64url_to_bytes(public_key),
        credential_current_sign_count=sign_count,
        require_user_verification=True,
    )
    return auth_verification.new_sign_count
```

**L4: What it checks:**
1. The `challenge` in the response matches the one stored in the session (prevents replay).
2. The `origin` (scheme+host+port) matches the Odoo server URL.
3. The `credential_public_key` in the response matches the stored public key for this credential ID.
4. The `sign_count` in the response is **greater than** the stored `sign_count` (prevents cloned authenticator replay).
5. `user_verification=REQUIRED` was honored by the authenticator (the UV flag in authenticatorData byte 33).

Returns the **new sign count** to store back to the DB, updating `sign_count`.

---

**`_get_session_challenge()`** — `@api.model` — Retrieves and consumes the session challenge

```python
def _get_session_challenge(self):
    challenge = request.session.pop('webauthn_challenge', None)
    if not challenge:
        raise AccessDenied('Cannot find a challenge for this session')
    return challenge
```

**Important:** Uses `pop()` — the challenge is **consumed after a single use**. This prevents replay attacks where an attacker intercepts the authentication response and re-submits it.

```python
def action_delete_passkey(self):
```

#### Methods

**`make_key(registration)`** — `@check_identity` — Finalize passkey creation

```python
def make_key(self, registration=None):
    assert registration, "registration can not be empty"
    self.ensure_one()

    verification = request.env['auth.passkey.key']._verify_registration_options(registration)

    self.env.user.write({'auth_passkey_key_ids': [Command.create({
        'name': self.name,
        'credential_identifier': bytes_to_base64url(verification['credential_id']),
    })]})

    passkey = self.env.user.auth_passkey_key_ids[0]
    self.env.cr.execute(SQL(
        "UPDATE auth_passkey_key SET public_key = %s WHERE id = %s",
        base64.urlsafe_b64encode(verification['credential_public_key']).decode(),
        passkey.id,
    ))
```

**L4: Why is public_key written via raw SQL?**
The `public_key` field has a compute method `_compute_public_key` (reads from DB column) and an inverse `_inverse_public_key` (no-op). The raw SQL `UPDATE` bypasses the ORM to write the raw bytes without triggering the compute. This is necessary because `verification['credential_public_key']` is a cryptography library key object, not a plain string.

The `credential_identifier` (base64url-encoded) is stored normally via ORM; the `public_key` is stored via direct SQL.

After creation: session token is recomputed to invalidate any existing sessions using stale tokens.

#### `_login(db, credential, user_agent_env)` — `@classmethod` — Passkey login

```python
@classmethod
def _login(cls, db, credential, user_agent_env):
    if credential['type'] == 'webauthn':
        webauthn = json.loads(credential['webauthn_response'])
        with Registry(db).cursor() as cr:
            cr.execute(SQL("""
                SELECT login
                  FROM auth_passkey_key key
                  JOIN res_users usr ON usr.id = key.create_uid
                 WHERE credential_identifier=%s
            """, webauthn['id']))
            res = cr.fetchone()
            if not res:
                raise AccessDenied(_('Unknown passkey'))
            credential['login'] = res[0]

    return super()._login(db, credential, user_agent_env=user_agent_env)
```

**L4: What this does for passkey login:**
1. Detects `type='webauthn'` in credentials.
2. Extracts the `credential_id` from the WebAuthn response JSON.
3. Direct SQL lookup: finds the `login` of the user who owns this credential ID.
4. Sets `credential['login']` to that login.
5. Delegates to `super()._login()` which creates the session.

This is a **username-less login** flow: the user only provides the passkey, and Odoo looks up their identity from the credential ID. The `rp_id` restriction ensures the passkey can only be used on the correct domain.

#### `_check_credentials(credential, env)` — In-session passkey verification

```python
def _check_credentials(self, credential, env):
    if credential['type'] == 'webauthn':
        webauthn = json.loads(credential['webauthn_response'])
        passkey = self.env['auth.passkey.key'].sudo().search([
            ("create_uid", "=", self.env.user.id),
            ("credential_identifier", "=", webauthn['id']),
        ])
        if not passkey:
            raise AccessDenied(_('Unknown passkey'))

        new_sign_count = self.env['auth.passkey.key']._verify_auth(
            webauthn, passkey.public_key, passkey.sign_count)

        passkey.sign_count = new_sign_count
        return {'uid': self.env.user.id, 'auth_method': 'passkey', 'mfa': 'skip'}

    else:
        return super()._check_credentials(credential, env)
```

Returns `{'auth_method': 'passkey', 'mfa': 'skip'}` — this signals the auth chain that passkey authentication counts as having completed MFA, so no additional factor is required.

#### `_get_session_token_fields()` / `_get_session_token_query_params()` — Cache Invalidation

```python
def _get_session_token_fields(self):
    return super()._get_session_token_fields() | {'auth_passkey_key_ids'}

def _get_session_token_query_params(self):
    params = super()._get_session_token_query_params()
    params['select'] = SQL("%s, ARRAY_AGG(key.id ORDER BY key.id DESC)", params['select'])
    params['joins'] = SQL("%s LEFT JOIN auth_passkey_key key ON res_users.id = key.create_uid",
                          params['joins'])
    return params
```

When a passkey is added or removed, the session token must be invalidated (recomputed). These methods ensure that the session token computation reads the passkey IDs, so any change to passkeys automatically invalidates the token.

## L4: WebAuthn Deep Dive

### WebAuthn Architecture

```
Browser (WebAuthn API)
    |
    | 1. navigator.credentials.create() with registration options
    |
    v
Authenticator (TouchID, FaceID, Windows Hello, YubiKey, etc.)
    |
    | 2. Generates key pair (private key stays on device)
    |    Stores credential_id + public_key + sign_count
    |
    v
Browser
    |
    | 3. Returns: attestationObject + clientDataJSON
    |
    v
Odoo Server (auth_passkey)
    |
    | 4. verify_registration_response() — stores public_key + credential_id
    |
    v
DB: auth_passkey_key
```

### Authentication Sequence

```
1. User visits /web/login, clicks "Login with Passkey"
2. JS calls GET /auth/passkey/start-auth
   → Server generates challenge, stores in session
   → Returns WebAuthn authentication options

3. Browser calls navigator.credentials.get(options)
   → Authenticator prompts for biometric/PIN
   → User verifies
   → Authenticator signs challenge with private key
   → Returns: authenticatorData + clientDataJSON + signature

4. JS POSTs to /web/login with type='webauthn', webauthn_response=JSON

5. Server:
   a. Looks up login from credential_id (SQL join)
   b. Verifies signature using stored public_key
   c. Checks sign_count increased
   d. Checks user_verification flag set
   e. Creates session
```

### Sign Count and Clone Detection

Each WebAuthn authenticator maintains an internal sign count that increments with every authentication. If Odoo sees a sign count LOWER than what it has stored, it indicates the authenticator has been cloned (an attacker copied the private key to another device). Odoo will reject such authentication.

Some software authenticators (e.g., KeePassXC) do not maintain a real sign count — Odoo tolerates this but cannot detect clones from such devices.

### User Verification (UV Flag)

In `_start_registration` and `_start_auth`, `user_verification=REQUIRED` is set. This means the authenticator MUST verify the user (via PIN, fingerprint, face, etc.) before completing the operation. The server independently verifies this by checking the UV bit in the returned `authenticatorData` byte 33.

Without UV, only "user presence" (the key is plugged in / nearby) is proven — not identity. Since passkey login skips MFA, Odoo requires UV as a 2FA-equivalent safeguard.

### RP ID Constraints

The `rp_id` (relying party ID) must be:
- The exact domain of the Odoo instance, OR
- A parent domain (e.g., `odoo.com` can cover `mycompany.odoo.com`)

`localhost` only works in development. In production, a valid domain is required — this prevents phishing sites on other domains from harvesting passkeys.

## L4: Passkey Storage Security

| Data | Stored In | Encrypted At Rest? |
|------|-----------|-------------------|
| Credential ID | `auth_passkey_key.credential_identifier` (DB) | No (not sensitive) |
| Public Key | `auth_passkey_key.public_key` (DB) | No (public key is not secret) |
| Private Key | **Never stored by Odoo** — stays on device | N/A |
| Session Challenge | `request.session['webauthn_challenge']` | Session store |

The private key is generated by and remains in the authenticator (OS keychain, hardware security module, or cloud sync). Odoo only stores the public half, which cannot be used to impersonate the user.

## Limitations and Known Gaps

| Issue | Description |
|-------|-------------|
| No cross-device login (by default) | Discoverable credentials (resident key) enable login without the original device, but most implementations sync via cloud (Apple iCloud Keychain, Google Password Manager). |
| `localhost` RP ID | Cannot register passkeys when Odoo runs on `localhost` in production. Requires a proper domain. |
| Hardware key tied to one domain | A passkey registered for `odoo.mycompany.com` cannot be used on `another.odoo.mycompany.com` unless `odoo.mycompany.com` is configured as the RP ID. |
| No passkey recovery flow | If all passkeys are lost and the password is unknown, an admin must reset the account. |
| LDAP coexistence | Passkeys and LDAP are independent auth mechanisms. A user can have both. |
